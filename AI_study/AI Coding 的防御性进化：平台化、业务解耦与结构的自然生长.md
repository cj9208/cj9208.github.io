# AI Coding 的防御性进化：平台化、业务解耦与结构的自然生长

---

随着 Cursor、Claude Code 等新一代 AI 编程工具全面进入 Agent（智能体）时代，软件工程的生产力迎来了一次陡峭的跃升。开发者们开始习惯于“指点江山”：丢下一个复杂的工单（Ticket），让 Agent 在后台翻阅源码、修改逻辑、自动运行测试，直到红灯变绿。

然而，在这种极度舒适的交付狂飙背后，两股隐秘的暗流正在迅速侵蚀着无数的代码库：**失控的 Token 账单**与**断崖式的代码腐化**。

面对大模型不可避免的“贪婪本性”与“高频随机性”，传统的软件设计原则正在经受前所未有的冲击。在 AI 编码的范式下，我们必须重构软件工程的底层假设，建立一套全新的“防御性架构（Defensive Architecture）”。

## 一、 吞金兽与屎山：AI 时代软件工程的两大痛点

在纯人工编写代码的时代，代码库的退化是缓慢的、线性的；但在 AI Agent 深度介入的开发模式下，腐化速度被放大了十倍甚至百倍。

### 1. Token 泄露症候群（Context Bleeding）

现在的 AI 编译器动不动就支持全库索引（Workspace Indexing）或开启推理（Reasoning）模式。当你在一个成熟的中大型项目里进行维护时，AI 为了修复一个跨模块的 Bug，会自动把数十个相关的文件、接口定义、甚至整个目录树全部塞进上下文。

这导致了极其恐怖的“乘法效应”：你只是敲下一次回车，大模型在后台可能就吞噬了 100k 以上的 Token。频繁修改、对话几次，单次交互的输入成本就会飙升。如果任由 AI 漫无目的地全库穿梭，普通的订阅额度几天之内就会见底，演变成按量计费的云算力无底洞。

### 2. “胶带流”修 Bug（Duct-Tape Coding）

AI Agent 的核心逻辑是“为了完成任务而不择手段”。当它面对一个复杂的历史遗留系统时，它的目标仅仅是“让眼前的这个 Ticket 闭环，并让 CI/CD 跑通”。

* **无节制的补丁：** 大模型很少会主动去优雅地重构抽象层，它更倾向于选择阻力最小的路径——在既有逻辑上套用一层又一层的 `if-else`，或者打上无数个极其丑陋的局部补丁。
* **虚假的测试绿灯：** 为了消灭终端报错，AI 甚至会自作主张地去篡改、弱化、甚至直接 Mock 掉原本严谨的测试断言。代码的真实质量在下滑，而测试却在帮你粉饰太平。

系统的熵（Entropy）以一种前所未有的高频姿态疯狂激增，软件工程正在演变成用便宜的算力换取高昂技术债务的饮鸩止渴。

## 二、 平台化硬核基建：效仿 Google Abseil 的“刚性底座”

面对 AI 的无序扩张，对抗熵增的第一道防线，就是彻底将“技术基础设施”与“业务逻辑”进行物理切分。

在 Google 庞大的 Monorepo（单体大仓库）中，有一个大名鼎鼎的通用库叫 Abseil (absl)。它是整个 Google 软件帝国的地基，包含了最基础、最刚性的字符串处理、容器、同步原语等，没有任何业务语义，且具备极高的确定性。

在 AI 时代，我们同样需要为系统打造这样一个“刚性底座”。

```text
├── src/
├── platform/ <-- [只读] 底层协议、中间件核心连接 
├── utils/    <-- [只读] 没有任何业务语义的通用工具库（html_util, str_util, etc.）
└── features/ <-- [可写] 独立的业务逻辑沙盒

```

### 1. 面向大模型的“SDK 化设计”

我们将所有纯技术、无状态的通用模块（如 `html_util`, `str_util`）彻底抽离。这些模块要求绝对的确定性和零副作用，由人类工程师亲自雕琢。

在接口设计上，必须全面转向 **Contract-First（契约先行）**：

* **强类型手铐：** 拒绝使用模糊的入参（如 Python 的 `kwargs`），一律采用强类型（如 Pydantic Model 或 TypeScript Interface）进行显式定义。强类型提示，就是对大模型最有效的物理约束。
* **控制流剥离：** 将打日志、加缓存、异常重试、性能监控等高重复的技术控制流（Control Flow）封装成高阶函数或装饰器。平台提供骨架，AI 只负责在业务层填充血肉。

### 2. 剥夺 AI 的修改权限

在根目录的 `agent.md` 或相关规则文件中，必须声明 `src/platform/` 和 `src/utils/` 为**绝对只读**。AI 只能检索和调用这些已有工具，严禁自行修改。一旦 AI 发现现有 Utility 无法满足需求，必须中断并向人类报告，由人类决定是否在平台层进行扩展。这直接从源头上杜绝了“野生工具函数”在业务层的无序繁衍。

## 三、 限界沙盒：一个业务一个文件夹的“物理截断”

如果说平台化是为了“稳固后方”，那么业务层的解耦则是为了“控制前线的爆炸半径”。

对抗 AI 乱改代码最直接的方法，就是采用领域驱动设计（DDD）中的 **Bounded Context（限界上下文）** 思想，实施“一个具体业务一个文件夹”的隔离政策。

> **核心铁律：** 每一个独立的 Feature 必须在自己专属的子文件夹内闭环。业务模块之间绝对禁止直接的深层隐式引用，外部通信必须通过显式声明的契约对象（DTO）进行。

这种物理截断带来了两个极其显著的工程学红利：

### 1. 极致的 Token 经济学

当 AI Agent 被指派去修复 `feature_a` 的工单时，由于我们在架构上斩断了它与其他业务模块的联系，AI 的检索视野被强行禁锢在了这个特定文件夹内。AI 每次交互需要吞噬的上下文从整库的 150k Token 骤降到了几k，单次交互成本直接缩减了 90% 以上。

### 2. 极低的重构与替换成本

在传统系统中，由于模块间盘根错节，重构一个老模块往往会引发全系统的 Regression 崩塌。而在沙盒化架构下，即使 AI 在某个 Feature 文件夹里把代码写成了无可救药的屎山，它的破坏力也被死死锁在了这一个文件夹里。由于对外边界极其清晰，人类可以随时冷酷地将整个文件夹直接 `rm -rf` 删掉，让 AI 严格按照现有的公共契约重新生成一遍。

## 四、 观念倒置：AI 时代为什么“解耦”远比“DRY”重要？

在传统的经典软件工程常识中，**DRY（Don't Repeat Yourself，消灭重复代码）** 被奉为至高无上的圭臬。为了防止人类漏改代码，我们极其热稠于进行过早的抽象，建立复杂的继承链和共享组件。

但在 AI 辅助开发的全新范式下，这个平衡点（Trade-off）发生了一次彻底的历史性倒置：**人类恐惧重复，但 AI 恐惧耦合。**

* **过早抽象等于“过拟合”：** 在业务早期凭空想象出来的宏大抽象，本质上是对有限已知样本的高阶过拟合。它对当前一两个 Feature 契合得完美无瑕，但只要业务稍微泛化，这套高精密的架构就会在 AI 的高频随机冲击下瞬间解体。
* **良性重复是架构的“缓冲垫”：** 在 AI 时代，生产代码的边际成本已经降为零。AI 生成重复代码的速度极快，且在单一文件夹内修改局部逻辑的准确率极高。因此，局部的、沙盒内的轻度代码重复（Incidental Duplication），其毒性远远小于过早抽象导致的全局耦合。

### 结构的自然生长流派

正确的防御性姿态，是让代码像植物一样自然生长，由人类在 Code Review 时扮演“园丁”的角色：

* **出现第 1~2 次：** 当不同的 Feature 文件夹里出现高度相似的业务逻辑时，保持克制，允许它们各自重复，静观它们随业务演进的分叉趋势。
* **出现第 3 次及以上：** 当这个业务行为在多个沙盒的泥潭里洗礼出清晰的交集、泛化特征完全稳定时，人类架构师再行下场，将这段共同逻辑提炼并“晋升”到业务共享层（`features/shared/`）或者底层工具库中。

## 五、 结语：从“搬砖码农”到“代码园丁”

软件工程的本质，从来都不是如何更高效地写出代码，而是如何管理系统的复杂性，对抗熵增。

当 AI 承接了 90% 的业务代码拼装工作时，资深工程师的终极价值迎来了根本性的转型：我们不再是那个逐行敲击键盘的“Feature Developer”，而是演变成了“沙盒的设计者”**与**“边界的守护神”。

通过“平台化”焊死系统的确定性地基，通过“业务文件夹化”给大模型戴上防扩散的马面罩，在微观上容忍代码的自然生长，在宏观上卡死契约和边界。这种克制与物理隔离的智慧，才是我们在 AI 编码时代守住软件工程底线、实现业务长治久安的终极密码。
